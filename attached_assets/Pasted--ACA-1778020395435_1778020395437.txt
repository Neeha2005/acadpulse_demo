You are building AcadPulse — a full-stack web application that aggregates
academic notifications from WhatsApp, Gmail, and Google Classroom, classifies
them using AI, extracts deadlines, scores urgency, and presents everything
in a clean dashboard with an agentic chatbot. Users are Pakistani university
students who communicate in both English and Roman Urdu.

Build the COMPLETE application end-to-end. Do not skip any section.
Follow the exact flow, features, and UI requirements described below.

Create these tables on startup if they don't exist:

users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  phone TEXT UNIQUE NOT NULL,
  email TEXT,
  university TEXT,
  password_hash TEXT NOT NULL,
  onboarding_complete BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT NOW()
)

platforms (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  platform TEXT NOT NULL,          -- 'whatsapp' | 'gmail' | 'classroom'
  connected BOOLEAN DEFAULT FALSE,
  tokens JSONB,                    -- Google OAuth tokens stored here
  session_data JSONB,              -- WhatsApp session state
  connected_at TIMESTAMP
)

courses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,              -- e.g. "Operating Systems"
  short_forms TEXT[],              -- e.g. ['OS', 'os', 'Op Sys']
  semester TEXT                    -- e.g. 'Fall 2025'
)

source_mappings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  course_id UUID REFERENCES courses(id) ON DELETE CASCADE,
  source_type TEXT NOT NULL,       -- 'whatsapp_group' | 'classroom' | 'gmail_thread'
  source_id TEXT NOT NULL,         -- WhatsApp JID, Classroom course ID, Gmail label
  source_name TEXT                 -- human-readable name
)

abbreviations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  abbreviation TEXT NOT NULL,
  full_form TEXT NOT NULL
)

notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  course_id UUID REFERENCES courses(id),
  source TEXT NOT NULL,            -- 'whatsapp' | 'gmail' | 'classroom' | 'manual'
  source_group TEXT,               -- group name or sender
  type TEXT NOT NULL,              -- 'Assignment'|'Quiz'|'Announcement'|'Material'|'Event'|'Exam Schedule'
  description TEXT NOT NULL,
  raw_message TEXT,
  deadline TIMESTAMP,
  urgency TEXT DEFAULT 'Low',      -- 'Low'|'Medium'|'High'|'Critical'|'Overdue'
  status TEXT DEFAULT 'pending',   -- 'pending'|'done'|'overdue'
  file_path TEXT,                  -- for materials auto-downloaded from WhatsApp
  file_url TEXT,                   -- for Classroom/Gmail direct links

  semester TEXT DEFAULT 'Fall 2025',

timetable (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,

  start_time TIME NOT NULL,

archived_semesters (

────────────────────
1. User opens the app URL

Sign Up tab collects:
  - Full name (text input)
  - Phone number (text input, with +92 prefix selector for Pakistan)
  - University name (text input)
  - Email address (optional)
  - Password + confirm password

On submit:
  - Validate all required fields client-side first
  - POST /api/auth/signup
  - Hash password with bcrypt, insert into users table
  - Return JWT token (24h expiry)
  - Store token in localStorage
  - Redirect → check onboarding_complete → if false → Onboarding page

  - Phone number or email
  - Password

On submit:
  - POST /api/auth/login
  - Verify credentials, return JWT
  - Store token in localStorage
  - Redirect → check onboarding_complete → if false → Onboarding, else → Dashboard

Error states to show clearly:
  - "Phone number already registered" on duplicate signup
  - "Incorrect phone number or password" on bad login
  - Field-level validation errors inline under each input

SECTION 5 — AI PIPELINE LOGIC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every incoming message (from any source) must pass through this pipeline:

STEP 1 — Classify
  Load fine-tuned XLM-RoBERTa model from /models/classifier
  If model not yet available (development), use a mock classifier that
  returns labels based on keyword matching as a fallback:
    Keywords like "assignment", "submit", "karo" → Assignment
    Keywords like "quiz", "test" → Quiz
    Keywords like "cancel", "venue", "shift" → Announcement
    Keywords like "slides", "notes", "upload" → Material
    Keywords like "event", "competition", "fair" → Event
    Keywords like "date sheet", "finals", "exam schedule" → Exam Schedule
    Anything else → Noise
  If label is Noise → stop pipeline, do not save to DB

STEP 2 — Identify course
  Check source_mappings table for the message's source group/ID
  If mapped → assign course_id
  If not mapped → send message text to Groq to guess the course:
    Known short forms: DL=Deep Learning, NLP=Natural Language Processing,
    DSA=Data Structures, OS=Operating Systems, DB=Database, OOP=Object Oriented
    Plus any stored in abbreviations table for this user
  If Groq confident → tag course, notify user in next dashboard load
  If ambiguous → save notification with course=null, show "Untagged" in UI,
    prompt user to assign: "We received a message we couldn't tag — please assign a course"

STEP 3 — Extract deadline (only for Assignment, Quiz, Exam Schedule)
  Send message text to Groq:
    System: "Extract any deadline from this academic message.
             Return JSON only: {\"has_deadline\": bool, \"deadline\": \"YYYY-MM-DD HH:MM\" or null}
             The message may be in English or Roman Urdu. Today is {today}.
             Roman Urdu time references: kal=tomorrow, aaj=today, aglay hafte=next week,
             raat=night (assume 23:59), subah=morning (assume 09:00)"
    User: {message text}
  Parse response JSON, save deadline field

STEP 4 — Score urgency
  def calculate_urgency(deadline, status):
    if status == 'done': return None
    if deadline is None: return 'Low'
    delta = deadline - datetime.now()
    if delta.total_seconds() < 0: return 'Overdue'
    if delta.days < 1: return 'Critical'
    if delta.days < 3: return 'High'
    if delta.days < 7: return 'Medium'
    return 'Low'

  Run this function on all pending notifications:
    - On every API call to GET /api/notifications
    - Via APScheduler every 3 hours in background
    - Immediately when a deadline is added or edited

STEP 5 — Save to DB and trigger notification if needed
  Insert/update notification row
  If urgency changed to High or Critical → trigger desktop popup via plyer
  If urgency is Critical → send WhatsApp self-message via Baileys bridge

SECTION 6 — CHATBOT IMPLEMENTATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

On every /api/chat POST:
1. Load the full user context from DB as JSON:
   {
     courses: [...],
     pending_notifications: [...],     // all with deadline, urgency, status
     timetable: [...],
     today: "YYYY-MM-DD",
     day_of_week: "Monday"
   }

2. Build Groq system prompt:
   "You are AcadPulse, an AI assistant for a Pakistani university student.
    You have access to the student's complete academic data shown below.
    You understand both English and Roman Urdu naturally.
    
    Your two modes:
    A) ANSWER: If user asks a question, answer from the data. Be specific.
       Include course names, deadlines, urgency levels.
    B) ACTION: If user gives an instruction, respond with a JSON action block:
       { \"action\": \"add_notification\" | \"mark_done\" | \"delete\" | \"add_timetable\" | \"update_mapping\",
         \"params\": { ... } }
    
    For DELETE actions always respond with:
       { \"action\": \"confirm_delete\", \"item\": \"...\", \"message\": \"Kya aap sure hain?\" }
    
    Common Roman Urdu you will see:
       karo/kar do = do it, kal = tomorrow, aaj = today, hafte = week,
       raat = night, submit = submit, pending = pending, done = done
    
    Student data: {json.dumps(context)}"

3. Call Groq with system prompt + full conversation history + new user message
4. Parse response:
   - If plain text → return as chat reply
   - If JSON action block → execute the action against DB → return confirmation message

SECTION 8 — UI DESIGN REQUIREMENTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DESIGN LANGUAGE:
  - Clean, modern, minimalist — inspired by Linear and Notion
  - Dark sidebar optional but use light background for main content
  - Primary color: deep indigo #4F46E5
  - Urgency colors: Critical=#EF4444 (red), High=#F97316 (orange),
                    Medium=#EAB308 (yellow), Low=#22C55E (green),
                    Overdue=#6B7280 (grey)
  - Course badges: auto-assign from a palette of 8 distinct soft colors
    (rotate through: blue, purple, teal, amber, rose, emerald, sky, violet)
  - Fonts: Inter (headings), system-ui (body)
  - Border radius: rounded-xl on cards, rounded-full on badges
  - Shadows: subtle (shadow-sm) — no heavy drop shadows
  - Spacing: generous padding, items should breathe

LOGIN/SIGNUP PAGE:
  - Full-page centered card on a subtle gradient background
    (indigo-50 to violet-50, or a very light mesh gradient)
  - AcadPulse logo + tagline at top: "Your academic life, organized."
  - Card has soft border and shadow
  - Tabs for Login / Sign Up with smooth animated underline

ONBOARDING:
  - Full-page with progress bar at very top (thin colored bar, width grows with steps)
  - Step number + title: "Step 1 of 4 — Connect your platforms"
  - Platform cards: large, icon + title + description, hover lifts slightly
  - Connected state shows a green checkmark badge on the card
  - Back and Continue buttons clearly at bottom

DASHBOARD NAVBAR:
  - White background, subtle bottom border
  - Logo left, tabs center (icon + label), user avatar right
  - Active tab has indigo underline
  - Unread count badge on the bell icon (red dot or number)

NOTIFICATION CARDS:
  - White cards with rounded-xl, shadow-sm
  - Left colored border (4px, urgency color)
  - Top row: course badge (pill) + type badge + source badge
  - Middle: description in medium-weight text
  - Bottom row: deadline text + relative time + mark-done button
  - Hover: card lifts slightly (translate-y-[-2px])
  - Done state: card greys out, description gets line-through

URGENCY BADGES:
  - Pill shaped, colored background matching urgency level
  - "🔴 Critical" / "🟠 High" / "🟡 Medium" / "🟢 Low" / "⚫ Overdue"

CHATBOT:
  - Right panel or full Chat tab
  - User messages: right-aligned, indigo background, white text
  - Bot messages: left-aligned, light grey background, dark text
  - Typing indicator: three animated dots
  - Timestamp on each message (small, muted)
  - Input bar: rounded-full, with paper plane send button

EMPTY STATES:
  - Every tab must have a pleasant empty state illustration (SVG inline)
  - Messages like "No pending tasks 🎉" or "No materials shared yet"

RESPONSIVE:
  - Works on laptop screens minimum
  - Tablet-friendly (tabs can collapse to icon-only on narrow)

LOADING STATES:
  - Skeleton loaders (not spinners) for card lists
  - Toast notifications (bottom-right) for actions: "Marked as done ✓"

SECTION 9 — BACKGROUND JOBS (APScheduler)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Schedule these jobs on FastAPI startup:

1. Urgency recalculation — every 3 hours
   For all users, recalculate urgency on all pending notifications
   If any change to Critical → trigger alert

2. Gmail poll — every 15 minutes
   For all users with Gmail connected
   Fetch new emails since last_fetched timestamp
   Run pipeline on each new email

3. Classroom poll — every 30 minutes
   For all users with Classroom connected
   Fetch new announcements and assignments

4. Daily morning digest — every day at 8:00 AM (local time)
   For each user:
     - Count today's deadlines, pending assignments, quizzes
     - Send WhatsApp message to user's own number:
       "🌅 AcadPulse Morning Digest\n
        Aaj ka summary:\n
        🔴 {n} critical deadlines today\n
        📝 {n} assignments pending\n
        📚 {n} quizzes this week\n
        Good luck! 🎯"

SECTION 10 — SEMESTER RESET
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

POST /api/semester/reset:
1. Show a confirmation modal: "This will archive your current semester and
   start fresh. Your data will be preserved under 'Fall 2025'. Continue?"
2. On confirm:
   a. Serialize all current notifications, timetable, mappings into JSONB
   b. Insert into archived_semesters with semester_label = current semester name
   c. Delete all notifications, timetable for this user
   d. Reset onboarding_complete = false (redirect to onboarding)
   e. Keep users, platforms, abbreviations tables intact
3. User can view past semesters from Settings → Archived Semesters

IMPORTANT NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Every page must handle loading, error, and empty states gracefully
- All API errors must return { error: string, detail: string } with correct HTTP status
- Never expose the JWT secret or API keys in frontend code
- Use .env file for all secrets — never hardcode
- Add CORS middleware in FastAPI allowing the React frontend origin
- WhatsApp: never auto-mark messages as read — just read silently
- All timestamps stored in UTC in DB, converted to user local time in frontend
- The XLM-RoBERTa model file is large — use the keyword-based mock classifier
  as default and only swap in the real model when explicitly loaded
- Run database migrations automatically on FastAPI startup using raw SQL
  (create tables IF NOT EXISTS)
- Use React Query or SWR for data fetching and cache management in frontend
- All destructive actions (delete, semester reset) require a confirmation dialog

Build the complete application now. Start with the file structure, then
implement each section in the priority order listed in Section 12.
